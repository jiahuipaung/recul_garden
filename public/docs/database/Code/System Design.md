# 相关理论
## 性能与可扩展性
如果服务**性能**的增长与资源的增加是成比例的，服务就是可扩展的。通常，提高性能意味着服务于更多的工作单元，另一方面，当数据集增长时，同样也可以处理更大的工作单位。
另一个角度来看待性能与可扩展性:
- 如果你的系统有**性能**问题，对于单个用户来说是缓慢的。
- 如果你的系统有**可扩展性**问题，单个用户较快但在高负载下会变慢。
## 延迟与吞吐量
**延迟**是执行操作或运算结果所花费的时间。
**吞吐量**是单位时间内（执行）此类操作或运算的数量。
通常，你应该以**可接受级延迟**下**最大化吞吐量**为目标。
## 可用性与一致性
### CAP理论
在发生网络分区（网络出现故障，导致节点间通信中断）的情况下，分布式系统只能同时满足以下三个条件中的两个
- **一致性** ─ 每次访问都能获得最新数据但可能会收到错误响应
- **可用性** ─ 每次访问都能收到非错响应，但不保证获取到最新数据
- **分区容错性** ─ 在任意分区网络故障的情况下系统仍能继续运行
**网络并不可靠，所以你应要支持分区容错性，并需要在软件可用性和一致性间做出取舍。**
##### 一致性模式
###### 弱一致性
在写入之后，访问可能看到，也可能看不到（写入数据）。尽力优化之让其能访问最新数据。
这种方式可以 memcached 等系统中看到。弱一致性在 VoIP，视频聊天和实时多人游戏等真实用例中表现不错。打个比方，如果你在通话中丢失信号几秒钟时间，当重新连接时你是听不到这几秒钟所说的话的。
###### 最终一致性


#### 核心结论：
- **一致性 + 可用性（CA）**：在没有网络分区的情况下，系统可以同时保证一致性和可用性。但当网络分区发生时，这两个条件中的一个必须牺牲。
- **一致性 + 分区容忍性（CP）**：如果系统要求在网络分区发生时仍保持一致性，则必须放弃可用性。也就是说，即使部分节点无法访问，系统依然保证一致的数据状态，无法接受响应的请求会被拒绝。
- **可用性 + 分区容忍性（AP）**：如果系统要求在网络分区时仍保持可用性，则必须放弃一致性。即使有些数据不同步，系统仍会提供服务。
#### 实际应用：
- **CA 系统**：例如，关系型数据库（如 MySQL）在没有分区的情况下可以保证一致性和可用性，但一旦出现网络分区，可能会丢失某些可用性或一致性保证。
- **CP 系统**：例如，Zookeeper。在网络分区的情况下，它优先保持一致性而非可用性。部分请求可能会失败，直到网络恢复正常。
- **AP 系统**：例如，Cassandra 或 Couchbase。在网络分区的情况下，它保证可用性，但可能会返回过时或不一致的数据，直到网络问题得到解决
