# 概览
服务单一业务、服务之间采用轻量级通信机制；
原子服务、独立进程、隔离部署（无法绕过代码逻辑访问其他服务的底层数据库）、去中心化服务治理
缺点：
对基础设施高度依赖、复杂度高；应用规模复杂也需要例如集中式的日志查看；
不可避免需要RPC或消息传递进行通信，因此也不得不编码处理消息传递过程中的速度问题、服务不可用等局部失效问题， **可用性工作**很重要

不同微服务使用独占的DB或者Cache，因此处理数据一致性问题也很重要。**分布式缓存、事务**

微服务组成：
- Kit：微服务的基础库例如gozero、Kratos
- service：业务代码+kit依赖+第三方依赖组成的业务微服务
- RPC+Message queue，轻量级通讯

发送时要保守（最小化传送必要的数据），接收时要开放（最大小洞容忍冗余数据）

# 设计
## API Gateway
客户端和后端服务之间的统一入口和代理层，既简化了客户端调用逻辑，也为后端服务提供了多种增强功能。
1. 若每个微服务直接暴露在外，客户端带微服务直接通信，会导致强耦合，需要一直维护旧API 
2. 需要多次请求，获取多个服务的数据，客户端聚合大量数据，工作量大、延迟高
3. 协议不利于统一，每个部门都有差异，需要端侧兼容
4. 端的API适配，耦合到了内部服务
5. 多终端兼容逻辑复杂，每个服务都需要处理
6. 统一逻辑无法收敛，比如安全验证，限流。只有对外暴露越少才能越收敛
BFF(backend for frontend)

实际业务流量：移动端->API Gateway -> BFF -> MircoService (忽略CDN和ELB)
### 原理
- **反向代理与路由**  
    API Gateway 作为反向代理服务器，接收所有外部请求，然后根据预设的路由规则，将请求转发给具体的后端服务或微服务。它会根据 URL 路径、请求头、认证信息等决定如何转发请求。
- **协议转换**  
    在微服务架构中，不同服务可能采用不同的通信协议（如 HTTP、WebSocket、gRPC 等），API Gateway 可以做协议转换，允许外部客户端使用统一的 API 协议，而后端服务可以使用适合自身的协议。
- **请求聚合与拆分**  
    对于一个业务请求可能需要调用多个微服务的场景，API Gateway 可以将多个服务的响应合并成一个统一的响应返回给客户端，减少客户端的多次调用。
    差异化服务：针对不同终端进行数据裁剪聚合、定制化
- **安全与认证**  
    API Gateway 通常集成了认证、授权、流量控制、日志记录等安全机制。它可以集中管理身份验证（如 OAuth、JWT 等），屏蔽内部微服务的直接暴露，提升整体安全性。
- **监控与限流**  
    通过对所有进出的请求进行监控，API Gateway 能够统计调用量、响应时间等指标，并结合限流、熔断等策略保护后端服务不被过载。
### 缺点
- **单点故障风险**  
    如果 API Gateway 本身没有做高可用设计，可能会成为整个系统的瓶颈或单点故障，影响所有服务的访问。
- **性能瓶颈与延迟**  
    所有请求都需经过网关处理，若处理不当或负载过高，可能引入额外延迟，影响系统响应速度。
- **复杂性增加**  
    随着功能不断增加（如安全、流控、聚合、协议转换等），API Gateway 的配置和维护会变得复杂，需要专业团队进行管理和监控。
- **耦合与灵活性问题**  
    虽然 API Gateway 能够统一管理多个微服务，但同时也可能成为系统各服务之间的耦合点，某些特殊场景下可能限制灵活性（例如在内部服务之间的直接高效调用）。

将应用程序划分为：命令端和查询端两部分。

## 安全
外网请求一般在API Gateway进行认证拦截，认证成功后使用Header方式通过RPC元数据传递的方式带到BFF层，BFF获取后将用户身份信息注入应用的Context中。BFF到下层微服务可以直接在RPC Request中带入用户身份信息

在服务内部，一般要区分身份认证(是谁)和授权（能调什么接口）
- Full Trust
- Half Trust
- Zero Trust







# 集中式的负载均衡和去中心化的负载均衡
两种不同的流量调度方式，其核心区别在于决策权的归属和实现方式。下面结合具体业务场景做详细说明：
### 1. 集中式负载均衡

**定义**  
集中式负载均衡是指通过一个或一组专门的负载均衡器（如硬件设备F5、软件方案Nginx、阿里云SLB等）来统一接收外部流量，然后根据预设的调度算法（例如轮询、最少连接、加权轮询等）将流量分发到后端服务器集群上。这种方式中，所有的流量都必须经过“网关”或“调度中心”，由其做出路由决策。

**业务场景举例**

- **网站访问入口**：对于电商网站，所有用户请求首先通过一个位于前端的集中式负载均衡器（例如云厂商提供的应用负载均衡器）。负载均衡器将流量根据当前服务器的负载情况或健康状况，分发到后端Web服务器集群中。这样，即使某台服务器出现故障，负载均衡器也能及时剔除故障节点，保证整体服务的稳定性。
    
- **数据中心出入口**：在一个数据中心中，所有对外的流量由专门的负载均衡器统一处理，既方便监控和管理，又可以在遇到大流量（如促销活动期间）时，实现快速扩展和故障转移。
    

**优点与缺点**

- **优点**：管理集中、监控和安全策略易于统一执行，能够提供透明的请求分发和故障隔离。
    
- **缺点**：可能成为单点瓶颈，尽管可以通过冗余设计来降低风险，但整体系统依赖于负载均衡器的稳定性和性能。
    

---

### 2. 去中心化负载均衡

**定义**  
去中心化负载均衡则是指负载均衡的决策不依赖于单一的集中式调度点，而是在客户端或各个服务内部实现。常见的方式是客户端负载均衡（如Netflix Ribbon、Spring Cloud LoadBalancer）：客户端从服务注册中心（如Eureka、Consul等）获取所有可用后端服务的列表，然后根据算法（比如随机、轮询、最少连接等）自行选择一个实例发起请求。这样，请求不再经过一个统一的负载均衡器，而是直接由客户端决定路由目标。

**业务场景举例**

- **微服务架构内部调用**：在微服务系统中，各个服务之间的调用通常采用去中心化的方式。比如，一个订单服务需要调用库存服务时，会从服务注册中心获取所有库存服务实例列表，然后由客户端根据一定的策略选择一个实例。这种方式可以减少中心节点的压力，提高系统的扩展性和容错性。
    
- **跨区域服务访问**：当不同区域的服务需要直接互相通信时，去中心化的负载均衡能使服务调用更加灵活，客户端可以根据网络延迟或地理位置选择最优节点，从而提升访问速度。
    

**优点与缺点**

- **优点**：消除了中心节点可能带来的瓶颈和单点故障问题；灵活性高，能够根据实时状态动态选择最优节点；在大规模微服务场景下能有效降低中间件压力。
    
- **缺点**：客户端需要维护服务列表和负载均衡逻辑，增加了客户端复杂度；各客户端决策可能不一致，统一的监控和安全控制也会变得更加复杂。
    

---

### 总结

- **集中式负载均衡**适合于对外部流量入口进行统一管理的场景，如传统网站、在线电商平台等，能提供统一的入口、简单的监控和安全策略，但需要关注可能的单点瓶颈问题。
    
- **去中心化负载均衡**则更适合于微服务架构内部的服务调用，允许各服务自主选择调用目标，减少中心节点压力，提高系统的弹性和扩展性，但需要额外在客户端实现负载均衡逻辑，并处理分布式环境下的一致性和安全问题。
    

这种选择通常需要根据实际业务需求、系统架构和性能指标来权衡。通过集中式与去中心化的合理结合，也可以构建出既具备统一管理能力，又能分散风险和提高灵活性的混合式负载均衡方案。
