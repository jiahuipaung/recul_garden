RPC（远程过程调用）之所以能实现跨语言通信和兼容性，主要基于以下几个底层设计原理：
### 1. 接口描述语言（IDL）
RPC 系统通常使用一种独立于语言的接口描述语言（IDL）来定义服务接口、数据结构和方法签名。通过 IDL 定义的接口是中立的，既不依赖于某一种编程语言，也不依赖于具体的实现细节。开发者使用 IDL 后，通过代码生成工具分别生成各自语言的客户端存根（stub）和服务端骨架（skeleton）。这样，无论客户端用哪种语言实现，只要遵循相同的接口定义，就能与使用其他语言实现的服务端进行通信。
### 2. 标准化的数据序列化与传输协议
RPC 框架（如 gRPC、Thrift、Protocol Buffers 等）采用标准化的数据序列化格式（例如 Protobuf、JSON、XML 等）以及统一的传输协议（通常基于 TCP/IP、HTTP/2 等）。这些序列化格式设计成与语言无关的二进制或者文本格式，使得不同语言之间能互相解析、序列化和反序列化数据，从而保证通信数据的一致性和兼容性。
### 3. 底层网络协议的普遍性
底层通信一般建立在 TCP/IP 这样全球通用的网络协议之上，而这些协议在所有现代编程语言中都有成熟的支持。RPC 系统正是利用了这一点，把网络传输细节抽象出来，提供一种像调用本地函数那样的编程体验，而不需要开发者关心底层的网络细节。
### 4. 抽象化的调用机制
RPC 的设计目标之一是使远程调用看起来像是本地调用。客户端通过调用本地的存根，实际上会自动进行参数的打包（marshalling）、网络传输、在服务端解包（unmarshalling）并调用真正的远程函数，然后将结果返回给客户端。这种抽象化机制将跨语言调用中的复杂性封装在框架内部，从而使不同语言之间的互操作变得透明和简单。
### 5. 版本兼容性设计
很多 RPC 框架在设计时都会考虑接口的版本演进问题，通过支持可选字段、默认值以及后向兼容的设计，使得在服务升级或接口扩展时，客户端和服务端之间依然能够保持兼容，确保不同版本之间能够正常通信。

---
总之，RPC 通过使用语言中立的接口定义、标准化的数据序列化格式和统一的底层网络协议，再加上对远程调用过程的抽象封装，使得不同编程语言开发的客户端与服务端能够无缝协作，从而实现了跨语言通信和兼容性。

# gRPC
- 多语言、语言中立；后端前端移动端
- 轻量级，高性能：序列化支持protobuf和json。PB是语言无关的高性能序列化框架
- 可插拔：插件
- IDL：基于文件定义服务，通过proto3工具生成指定语言的数据接口、服务端代码以及客户端stub。requst和response
- 移动端支持：基于HTTP 2.0设计，支持双向流，消息头压缩、单TCP的多路复用、服务端推送等特性

- 服务而非对象，消息而非引用：促进微服务的系统间粗粒度消息交互设计理念
	- 重点放在独立的服务及其交互上，而不是把分布式系统看作是一堆可以直接远程调用的对象。
	- 试图像本地调用对象时使用细粒度方法会增加系统耦合和延迟，因此力度要粗，只关注整个服务提供的功能
	- 远程调用过程中，不再传递对某个对象的引用，而是将所需的数据封装成消息（通常使用 Protocol Buffers 进行序列化），一次性传递给对方。
	- 一次调用会传递更多的上下文信息，减少了需要多次跨网络调用的可能，从而减少了网络往返次数和潜在的延迟问题。
- 负载无关：不同服务需要使用不同类型的消息编码：json、xml、pb、Thrift
- Streaming API
- 阻塞式、非阻塞式：支持同步异步处理
- 元数据交换：常见的认证追踪（IP traceID等等），进行限流熔断时需要
- 使用了标准化状态码

不要过早关注性能，先进行标准化
## health check
当服务不稳定时，消费者可感知，并从负载均衡池中剔除；check成功后恢复

## 服务发现
- 客户端发现
	- 客户端负责确定可用服务实例的网络位置和请求负载均衡
	- 缺点是它将客户端与服务注册中心耦合在一起。你必须为你使用的每种编程语言和框架实现客户端服务发现逻辑
- 服务端发现
	- 客户端通过负载均衡器向服务发出请求。负载均衡器查询服务注册中心并将每个请求路由到可用的服务实例
