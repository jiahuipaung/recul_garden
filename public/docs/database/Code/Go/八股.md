## 通道及底层原理理解
- 通道分为三种类型：双向，单向发送（chan<- T），单向接收（<-chan T），有容量属性，表示是否有缓冲
- 随着一个数据值的传递（发送和接收），一些数据值的所有权从一个协程转移到了另一个协程。 当一个协程发送一个值到一个通道，我们可以认为此协程释放了（通过此发送值可以访问到的）一些值的所有权。 当一个协程从一个通道接收到一个值，我们可以认为此协程获取了（通过此接受值可以访问到的）一些值的所有权。
- 当然，在通过通道传递数据的时候，也可能没有任何所有权发生转移。
- 所有通道类型均为**可比较**类型。

从[值部](https://gfw.go101.org/article/value-part.html)一文，我们了解到一个通道值可能含有底层部分。 当一个通道值被赋给另一个通道值后，这两个通道值将共享相同的底层部分。 换句话说，这两个通道引用着同一个底层的内部通道对象。 比较这两个通道的结果为`true`。

一个**数据接收操作 <-** 可以被认为是一个单值表达式。 但是，当一个数据接收操作被用做一个赋值语句中的唯一的源值的时候，它可以返回第二个可选的类型不确定的布尔值返回值从而成为一个多值表达式。 此类型不确定的布尔值表示**第一个返回值是否是在通道被关闭之前被发送的**。 sentBeforeClosed

### 内部实现
一个通道内部维护了三个队列（均可被视为先进先出队列）：
1. **接收数据协程**队列（可以看做是先进先出队列但其实并不完全是，见下面解释）。此队列是一个**没有长度限制**的链表。 此队列中的协程均处于**阻塞状态**，它们正等待着从此通道接收数据。
2. **发送数据协程**队列（可以看做是先进先出队列但其实并不完全是，见下面解释）。此队列也是一个**没有长度限制**的链表。 此队列中的协程亦均处于**阻塞状态**，它们正等待着向此通道发送数据。 此队列中的每个协程将要**发送的值**（或者此值的指针，取决于具体编译器实现）和**此协程**一起存储在此队列中。
3. **数据缓冲**队列。这是一个循环队列（**绝对先进先出**），它的长度为此通道的容量。此队列中存放的值的类型都为此通道的元素类型。 如果此队列中当前存放的值的个数已经达到此通道的容量，则我们说此通道已经处于满槽状态。 如果此队列中当前存放的值的个数为零，则我们说此通道处于空槽状态。 对于一个非缓冲通道（容量为零），它总是同时处于满槽状态和空槽状态。
每个通道内部维护着一个互斥锁用来在各种通道操作中防止数据竞争。
一个通道的内部的三个队列的各种事实：
- 如果一个通道已经关闭了，则它的发送数据协程队列和接收数据协程队列肯定都为空，但是它的缓冲队列可能不为空。
- 在任何时刻，如果缓冲队列不为空，则接收数据协程队列必为空。
- 在任何时刻，如果缓冲队列未满，则发送数据协程队列必为空。
- 如果一个通道是缓冲的，则在任何时刻，它的发送数据协程队列和接收数据协程队列之一必为空。
- 如果一个通道是非缓冲的，则在任何时刻，一般说来，它的发送数据协程队列和接收数据协程队列之一必为空， 但是有一个例外：一个协程可能在一个`select流程控制`中同时被推入到此通道的发送数据协程队列和接收数据协程队列中。
在一个值被从一个协程传递到另一个协程的过程中，此值将被复制至少一次
### Select
在一个`select-case`流程控制中，
- `select`关键字和`{`之间不允许存在任何表达式和语句。
- `fallthrough`语句不能被使用.
- 每个`case`关键字后必须跟随一个通道接收数据操作或者一个通道发送数据操作。 通道接收数据操作可以做为源值出现在一条简单赋值语句中。 以后，一个`case`关键字后跟随的通道操作将被称为一个`case`操作。
- 所有的非阻塞`case`操作中将有一个被随机选择执行（而不是按照从上到下的顺序），然后执行此操作对应的`case`分支代码块。
- 在所有的`case`操作均为阻塞的情况下，如果`default`分支存在，则`default`分支代码块将得到执行； 否则，当前协程将被推入所有阻塞操作中相关的通道的发送数据协程队列或者接收数据协程队列中，并进入阻塞状态。
### `select-case`流程控制的实现机理
`select-case`流程控制是Go中的一个重要和独特的特性。 下面列出了官方标准运行时中`select-case`流程控制的[实现步骤](https://github.com/golang/go/blob/master/src/runtime/select.go)。
1. 将所有`case`操作中涉及到的通道表达式和发送值表达式按照从上到下，从左到右的顺序一一估值。 在赋值语句中做为源值的数据接收操作对应的目标值在此时刻不需要被估值。
2. 将所有分支随机排序。`default`分支总是排在最后。 所有`case`操作中相关的通道可能会有重复的。
3. 为了防止在下一步中造成（和其它协程互相）死锁，对所有`case`操作中相关的通道进行排序。 排序依据并不重要，官方Go标准编译器使用通道的地址顺序进行排序。 排序结果中前`N`个通道不存在重复的情况。 `N`为所有`case`操作中涉及到的不重复的通道的数量。 下面，_**通道锁顺序**_是针对此排序结果中的前`N`个通道来说的，_**通道锁逆序**_是指此顺序的逆序。
4. 按照上一步中的生成通道锁顺序获取所有相关的通道的锁。
5. 按照第_2_步中生成的分支顺序检查相应分支：
    1. 如果这是一个`case`分支并且相应的通道操作是一个向关闭了的通道发送数据操作，则按照通道锁逆序解锁所有的通道并在当前协程中产生一个恐慌。 跳到第_12_步。
    2. 如果这是一个`case`分支并且相应的通道操作是非阻塞的，则按照通道锁逆序解锁所有的通道并执行相应的`case`分支代码块。 （此相应的通道操作可能会唤醒另一个处于阻塞状态的协程。） 跳到第_12_步。
    3. 如果这是`default`分支，则按照通道锁逆序解锁所有的通道并执行此`default`分支代码块。 跳到第_12_步。
6. （到这里，`default`分支肯定是不存在的，并且所有的`case`操作均为阻塞的。）
7. 将当前协程（和对应`case`分支信息）推入到每个`case`操作中对应的通道的发送数据协程队列或接收数据协程队列中。 当前协程可能会被多次推入到同一个通道的这两个队列中，因为多个`case`操作中对应的通道可能为同一个。
8. 使当前协程进入阻塞状态并且按照通道锁逆序解锁所有的通道。    
9. ...，当前协程处于阻塞状态，等待其它协程通过通道操作唤醒当前协程，...
10. 当前协程被另一个协程中的一个通道操作唤醒。 此唤醒通道操作可能是一个通道关闭操作，也可能是一个数据发送/接收操作。 如果它是一个数据发送/接收操作，则（当前正被解释的`select-case`流程中）肯定有一个相应`case`操作与之配合传递数据。 在此配合过程中，当前协程将从相应`case`操作相关的通道的接收/发送数据协程队列中弹出。
11. 按照第_3_步中的生成的通道锁顺序获取所有相关的通道的锁。
12. 将当前协程从各个`case`操作中对应的通道的发送数据协程队列或接收数据协程队列中（可能以非弹出的方式）移除。
    1. 如果当前协程是被一个通道关闭操作所唤醒，则跳到第_5_步。
    2. 如果当前协程是被一个数据发送/接收操作所唤醒，则相应的`case`分支已经在第_9_步中知晓。 按照通道锁逆序解锁所有的通道并执行此`case`分支代码块。
13. 完毕。
从此实现中，我们得知
- 一个协程可能同时多次处于同一个通道的发送数据协程队列或接收数据协程队列中。
- 当一个协程被阻塞在一个`select-case`流程控制中并在以后被唤醒时，它可能会从多个通道的发送数据协程队列和接收数据协程队列中被移除。
## 值部
在C中，每个值在内存中只占据一个[内存块](https://gfw.go101.org/article/memory-block.html)（一段连续内存）；但是，一些Go类型的值可能占据多个内存块。
我们称一个Go值分布在不同内存块上的部分为此值的各个值部（value part）。 一个分布在多个内存块上的值含有一个直接值部和若干被此直接值部[引用着](https://gfw.go101.org/article/pointer.html#references)的间接值部。

注意：
- 接口类型和字符串类型值是否包含间接部分取决于具体编译器实现。 如果不使用今后将介绍的非类型安全途径，我们无法从这两类类型的值的外在表现来判定它们的值是否含有间接部分。 在《Go语言101》中，我们认为这两类类型的值是可能包含间接值部的。
- 同样地，函数类型的值是否包含间接部分几乎也是不可能验证的。 在《Go语言101》中，我们认为函数值是可能包含间接值部的。

## Panic和Recover原理
**一般用恐慌表示正常情况下不应出现的逻辑错误**
我们可以调用内置函数`panic`来产生一个恐慌以使当前协程进入恐慌状况。
进入恐慌状况是另一种使当前函数调用开始返回的途径。 一旦一个函数调用产生一个恐慌，此函数调用将立即进入它的退出阶段。
通过在一个延迟函数调用之中调用内置函数`recover`，当前协程中的一个恐慌可以被消除，从而使得当前协程重新进入正常状况。
如果一个协程在恐慌状况下退出，它将使整个程序崩溃。
内置函数`panic`和`recover`的声明原型如下：
```Go
func panic(v interface{})
func recover() interface{}
```
接口（interface）类型和接口值将在以后的文章[接口](https://gfw.go101.org/article/interface.html)中详解。 目前，我们可以暂时将空接口类型`interface{}`视为很多其它语言中的`any`或者`Object`类型。 换句话说，在一个`panic`函数调用中，我们可以传任何实参值。
一个`recover`函数的返回值为其所恢复的恐慌在产生时被一个`panic`函数调用所消费的参数。
### 函数调用的退出阶段
在Go中，一个函数调用在其退出完毕之前可能将经历一个退出阶段。 在此退出阶段，所有在执行此函数调用期间被推入延迟调用队列的延迟函数调用将按照它们的推入顺序的逆序被执行。 当这些延迟函数调用都退出完毕之后，此函数调用的退出阶段也就结束了，或者说此函数调用也退出完毕了。
退出阶段有时候也被称为返回阶段。
一个函数调用可能通过三种途径进入它的退出阶段：
1. 此调用正常返回；
2. 当此调用中产生了一个恐慌；
3. 当`runtime.Goexit`函数在此调用中被调用并且退出完毕。
一般`runtime.Goexit()`函数不希望在主协程中调用，且是不能被取消的。但panic可以被恢复
### 函数调用关联恐慌和Goexit信号
当一个**函数调用**中直接产生了一个恐慌的时候，我们可以认为此（尚未被恢复的）恐慌将和此函数调用相关联起来。 类似地，当一个函数调用直接调用了`runtime.Goexit`函数，则`runtime.Goexit`函数返回完毕之后，我们可以认为一个Goexit信号将和此函数调用相关联起来。
一个函数调用最多只能和一个未恢复的恐慌相关联。 如果一个调用正和一个未恢复的恐慌相关联，则
- 在此恐慌被恢复之后，此调用将不再和任何恐慌相关联。
- 当在此函数调用中产生了一个新的恐慌，此新恐慌将替换原来的未被恢复的恐慌做为和此函数调用相关联的恐慌。
因为Goexit信号不可被取消，争论一个函数调用是否最多只能和一个Goexit信号相关联是没有意义和没有必要的。
在某个时刻，一个**协程**中可能**共存多个**未被恢复的恐慌，协程入口函数调用可能已产生panic，当进入延迟调用时，其可能又关联了一个新的panic
类似地，当一个和Goexit信号相关联的内层函数调用退出完毕后，此Goexit信号也将传播到外层函数调用中，并和外层函数调用相关联起来。 如果外层函数调用尚未进入退出阶段，则其将立即进入。
当一个Goexit信号和一个函数调用相关联起来的时候，如果此函数调用正在和一个未被恢复的恐慌相关联着，则此恐慌将被恢复。
内置`recover`函数必须在合适的位置调用才能发挥作用；否则，它的调用相当于空操作。
在下面的情况下，`recover`函数调用的返回值为`nil`：
- 传递给相应`panic`函数调用的实参为nil；
- 当前协程并没有处于恐慌状态；
- `recover`函数并未**直接在一个延迟函数调用中调用，** 或者它直接被一个延迟函数调用所直接调用但是此延迟调用没有被和期望被恢复的恐慌相关联的函数调用所直接调用。（两者都需要与对应的函数调用相关联）
> 一个recover调用只有在它的直接外层调用（即recover调用的父调用）是一个延迟调用，并且此延迟调用（即父调用）的直接外层调用（即recover调用的爷调用）和当前协程中最新产生并且尚未恢复的恐慌相关联时才起作用。 一个有效的recover调用将最新产生并且尚未恢复的恐慌和与此恐慌相关联的函数调用（即爷调用）剥离开来，并且返回当初传递给产生此恐慌的panic函数调用的参数。

    1. 数组是固定大小的，长度是数组类型的一部分，而Slice是动态大小的，可以变化。Slice是对数组的一个视图，包含指向底层数组的指针、长度和容量。

    1. 当Slice被复制或传递时，只会进行浅拷贝，复制的是指向底层数组的指针，而不是底层数组的内容。因此，修改新Slice的元素会影响到原Slice。

    1. `sync.Map`是为了高并发场景设计的，底层使用了分段锁和原子操作。它使用`unsafe.Pointer`和`atomic`包来实现高效的并发读写。
9. **sync.Mutex的底层实现**：
    1. `sync.Mutex`的实现基于操作系统提供的低级同步原语，如互斥锁。具体实现因操作系统而异，通常会使用系统调用来进行加锁和解锁。

13. **GMP模型**：
    
    1. Go的GMP模型代表Goroutine、Machine和Processor。Goroutine是go协程，Machine代表操作系统线程，Processor是调度的单位。通过将Goroutine分配给Processor，Go实现了高效的并发执行。
        
14. **IO密集型和计算密集型协程影响**：
    
    1. 在GMP模型中，IO密集型协程通常会被调度到空闲的Processor上，而计算密集型协程可能会占用Processor的时间片，导致IO协程的响应时间延长。
        
15. **GMP模型的优缺点**：
    
    1. 优点：调度开销小、上下文切换快。缺点：在极端情况下，可能会导致Goroutine过多，造成资源耗尽。

    1. gRPC的实现层次包括传输层（如HTTP/2）、序列化层（Protocol Buffers）、以及服务层（服务接口定义和调用）。

# 逃逸分析
1. 从性能的角度出发，在**栈上**分配内存和在**堆上**分配内存，**性能差异**是非常大的。在函数中申请一个对象，如果分配在**栈中**，函数执行结束时**自动回收**，如果分配在**堆中**，则在函数结束后某个时间点进行**垃圾回收**。在栈上分配和回收内存的开销很低，只需要 2 个 CPU 指令：PUSH 和 POP，一个是将数据 push 到栈空间以完成分配，pop 则是释放空间，也就是说在栈上分配内存，消耗的仅是将数据拷贝到内存的时间，而内存的 I/O 通常能够达到 30GB/s，因此在栈上分配内存效率是非常高的。在堆上分配内存，一个很大的额外开销则是**垃圾回收**。Go 语言使用的是**标记清除算法**，并且在此基础上使用了**三色标记法和写屏障技术**，提高了效率。
2. 什么是逃逸分析？
	1. 堆内存是通过垃圾回收机制自动管理的，无需开发者指定。那么，Go 编译器怎么知道某个变量需要分配在栈上，还是堆上呢？**go编译器决定内存分配位置的方式**，就称之为逃逸分析(escape analysis)。逃逸分析由编译器完成，作用于编译阶段。
	2. 变量逃逸的几种场景：
		1. 局部变量作为返回值返回到函数外部
		2. 变量被传递到其他 Goroutine，不再受当前栈的控制
		3. 取了变量的地址并存储到堆中
		4. 局部变量被赋值给一个接口类型的变量
3. **指针逃逸**是最容易理解的一种情况，即在函数中创建了一个对象，返回了这个对象的指针。这种情况下，函数虽然退出了，但是**因为指针的存在，对象的内存不能随着函数结束而回收，因此只能分配在堆上**。（编译时可以借助选项 `-gcflags=-m`，查看变量逃逸的情况）
4. i**nterface{} 动态类型逃逸：** 空接口即 `interface{}` 可以表示任意的类型，如果函数参数为 `interface{}`，编译期间很难确定其参数的具体类型，也会发生逃逸。因为 `fmt.Println()` 的参数类型定义为 `interface{}`，因此也发生了逃逸。
5. **栈空间不足**：操作系统对内核线程使用的栈空间是有大小限制的，64 位系统上通常是 8 MB。可以使用 `ulimit -a` 命令查看机器上栈允许占用的内存的大小。因此递归函数实现不当时，容易导致栈溢出。对于 Go 语言来说，运行时(runtime) 尝试**在 goroutine 需要的时候动态地分配栈空间**，goroutine 的初始栈大小为 2 KB。当 goroutine 被调度时，会绑定内核线程执行，栈空间大小也不会超过操作系统的限制。超过一定大小的局部变量将逃逸到堆上，或者无法确定长度时，将分配到堆上。
6. **闭包**：一个函数和对其周围状态（lexical environment，词法环境）的引用捆绑在一起（或者说函数被引用包围），这样的组合就是闭包（closure）。**一个携带了它的作用域环境的函数**。也就是说，闭包让你可以在一个内层函数中访问到其外层函数的作用域。例如一个A返回一个闭包函数，该函数访问量A中声明的一个局部变量n，A函数返回值赋给IN，那么A函数退出之后，变量n所占的内存当然不能释放，因此会逃逸到堆上。捕获的变量是 **引用**，不是值。修改捕获的变量会影响所有引用它的闭包。
7. 利用逃逸分析提升性能：传值会拷贝整个对象，而传指针只会拷贝指针地址，指向的对象是同一个。传指针可以减少值的拷贝，但是会导致内存分配逃逸到堆中，增加垃圾回收(GC)的负担。在对象频繁创建和删除的场景下，**传递指针导致的 GC 开销可能会严重影响性能**。一般情况下，对于需要修改原对象值，或占用内存比较大的结构体，选择传指针。对于只读的占用内存较小的结构体，直接传值能够获得更好的性能。
# GC垃圾回收
1. 最常见的垃圾回收算法有**标记清除(Mark-Sweep)** 和**引用计数(Reference Count)**，Go 语言采用的是**标记清除算法**。并在此基础上使用了**三色标记法和写屏障技术**，提高了效率
2. 标记清除收集器是跟踪式垃圾收集器，其执行过程可以分成标记（Mark）和清除（Sweep）两个阶段：
        1. 标记阶段 — 从**根对象**出发查找并**标记堆中所有存活的对象**；    
        2. 清除阶段 — 遍历堆中的全部对象，回收未被标记的垃圾对象并将回收的内存加入空闲链表。  
4. 标记清除算法的一大问题是在标记期间，需要暂停程序（Stop the world，STW），标记结束之后，用户程序才可以继续执行。为了能够异步执行，减少 STW 的时间，Go 语言采用了三色标记法。
5. 赋值器和回收器
6. 根对象在垃圾回收的术语中又叫做根集合，它是垃圾回收器在标记过程时最先检查的对象，包括：1. **全局变量**：程序在编译期就能确定的那些存在于程序整个生命周期的变量。2. **执行栈**：每个 goroutine 都包含自己的执行栈，这些执行栈上包含栈上的变量及**指向分配的堆内存区块的指针**。3. **寄存器**：寄存器的值可能表示一个指针，参与计算的这些指针可能指向某些赋值器分配的堆内存区块。
7. 常见的GC算法包括：1. 追踪式，分为多种不同类型，例如：标记清扫、标记整理、增量式、增量整理、分代式；2. 引用计数；
8. Go 的 GC 目前使用的是**无分代**（对象没有代际之分）、**不整理**（回收过程中不对对象进行移动与整理）、**并发**（与用户代码并发执行）的三色标记清扫算法。原因：1. **分代无用**，Go 的编译器会通过**逃逸分析**将大部分新生对象存储在栈上（栈直接被回收），只有那些需要长期存在的对象才会被分配到需要进行垃圾回收的堆中。2. 无需整理：Go 运行时的分配算法基于 tcmalloc，基本上没有碎片问题。 并且顺序内存分配器在多线程的场景下并不适用。Go 使用的是基于 tcmalloc 的现代内存分配算法，对对象进行整理不会带来实质性的性能提升
9. 三色：白（可能死亡）、灰（波面）、黑（确认存活）
10. STW：`stop the word` 到`Start the World` 这一动作发生时这一段时间间隔，即万物静止；STW 在垃圾回收过程中为了保证实现的正确性、防止无止境的内存增长等问题而不可避免的需要**停止赋值器**进一步操作对象图的一段过程
11. 调试方式：
12. 为什么有GC还会内存泄漏？预期的能很快被释放的内存由于附着在了长期存活的内存上、或生命期意外地被延长，导致预计能够立即回收的内存而长时间得不到回收。    
        1. 形式1：预期能被快速释放的内存因**被根对象引用**而没有得到迅速释放    
        2. 形式2：goroutine 泄漏：一个程序持续不断地产生新的 goroutine、且不结束已经创建的 goroutine 并复用这部分内存（维护执行用户代码的上下文信息）这种形式的 goroutine 泄漏还可能**由 channel 泄漏导致**。而 channel 的泄漏本质上与 goroutine 泄漏存在直接联系。Channel 作为一种同步原语，会连接两个不同的 goroutine，如果一个 goroutine 尝试**向一个没有接收方的无缓冲 channel 发送消息**，则该 goroutine 会被永久的休眠，整个 goroutine 及其执行栈都得不到释放
13. 并发标记清除法的难点是什么？并发回收的根本问题在于，用户态代码在回收过程中会**并发地更新**对象图，造成赋值器和回收器可能对对象图的结构产生不同的认知。保证标记与清除过程的正确性（**不应出现对象的丢失，也不应错误的回收还不需要回收的对象**）
14. 什么是写屏障、混合写屏障？如何实现？   
        1. 破坏正确性的两个必要条件：
            1. **条件 1**: 赋值器修改对象图，导致某一黑色对象引用白色对象；
            2. **条件 2**: 从灰色对象出发，到达白色对象的、未经访问过的路径被赋值器破坏。
        2. 避免其中任何一个条件，则不会出现对象丢失的情况            
        3. 将三色不变性所定义的波面根据这两个条件进行削弱        
            1. 当满足原有的三色不变性定义（或上面的两个条件都不满足时）的情况称为**强三色不变性（strong tricolor invariant）**        
            2. 当赋值器令黑色对象引用白色对象时（满足条件 1 时）的情况称为**弱三色不变性（weak tricolor invariant）弱三色不变形的好处在于：只要存在未访问的能够到达白色对象的路径，就可以将黑色对象指向白色对象**        
        4. 当赋值器进一步破坏灰色对象到达白色对象的路径时（**进一步满足条件 2** 时），即打破弱三色不变性， 也就破坏了回收器的正确性；   
        5. 通过赋值器屏障技术来保证指针的读写操作一致。使赋值器在进行指针写操作时，能够“通知”回收器，进而不会破坏**弱三色不变性**。实质就是在运行期间拦截内存写操作，相当于一个 hook 调用，阻止一些事情的发生，或者说做好一些标记工作，从而保证垃圾回收的正确性    
        6. 三种屏障技术：插入、删除、混合；区别就是 hook 的时机不同；是编译器生成的

### 数组与切片：
1. slice与数组的区别：
	1. slice是结构体，其中底层数组是地址
	2. slice切片操作、append操作（若不扩容）均共享底层数组，copy时如果不新建且容量够，则也会共享底层数组；结构体中包含切片时也可能共享
3. 容量增长方式：
	1. 由append函数追加元素，引发扩容，可追加元素值或直接追加切片；且需要接受返回值
	2. 当原 slice 容量小于 `1024` 的时候，新 slice 容量变成原来的 `2` 倍；原 slice 容量超过 `1024`，新 slice 容量变成原来的`1.25`倍。（golang1.18版本更新之前）
	3. 当原slice容量(oldcap)小于256的时候，新slice(newcap)容量为原来的2倍；原slice容量超过256，新slice容量newcap = oldcap+(oldcap+3 * 256)/4   （1.18版本更新之后）
- **Go 1.9.5：**
	    - 从 `len >= 1024` 时突然切换到 **1.25 倍**增长，没有过渡机制。
	    - 导致当容量刚好接近 1024 时，扩容的跳变可能会显得不自然。
```go

		newcap = cap
	} else {
		if old.len < 1024 {
			newcap = doublecap
		} else {
			for newcap < cap {
				newcap += newcap / 4
			}
		}
	}
```

- **Go 1.18：**
	    - 扩容过渡更加平滑。对于 `cap >= 256` 的切片，通过公式 `(newcap + 3*threshold) / 4`，让扩容在小于 2 倍时逐渐靠近 1.25 倍增长。
```go
// go 1.18 src/runtime/slice.go:178
func growslice(et *_type, old slice, cap int) slice {
    // ……
    newcap := old.cap
	doublecap := newcap + newcap
	if cap > doublecap {
		newcap = cap
	} else {
		const threshold = 256
		if old.cap < threshold {
			newcap = doublecap
		} else {
			for 0 < newcap && newcap < cap {
                // Transition from growing 2x for small slices
				// to growing 1.25x for large slices. This formula
				// gives a smooth-ish transition between the two.
				newcap += (newcap + 3*threshold) / 4
			}
			if newcap <= 0 {
				newcap = cap
			}
		}
	}
	// ……
    
	capmem = roundupsize(uintptr(newcap) * ptrSize)
	newcap = int(capmem / ptrSize)
}
```
此处第三个参数cap是新 slice 最小求的容量

### map 哈希表
map 是由 `key-value` 对组成的；`key` 只会出现一次；操作包括增、删、查、改等基本操作。
**哈希表**：Go 的 map 底层使用哈希表来存储数据。哈希函数将 `key` 映射到一个桶（bucket），同一个桶内的元素会根据哈希值分配到不同的位置。
**哈希冲突**：哈希冲突指不同的 `key` 被映射到同一个桶中，常用的解决办法是链表法和开放地址法。
#### 创建方式
- 未初始化的 `map` 为 `nil`，无法向其中添加键值对，会引发运行时错误。`nil` map 无法插入值，空 map 可以插入值。
- 必须使用 `make` 或字面量来初始化。
- 使用 `make` 函数创建 map，底层调用 `makemap` 函数来初始化 `hmap` 结构体，包括设置哈希种子、分配桶等。
##### 使用 `make` 创建：
```go
ageMap := make(map[string]int)      // 创建一个空 map
ageMapWithCapacity := make(map[string]int, 10) // 带初始容量的 map
```
##### 字面两创建
```go
person := map[string]int{"Alice": 25, "Bob": 30}
```
##### 声明但未初始化
```go
var m map[string]int // 未初始化，值为 nil
fmt.Println(m == nil) // true
```


#### 内存模型
##### **实现结构**：
- **hmap**：map 底层是 `hmap` 结构体，包含一些关于哈希表的信息，如桶数组 `buckets` 和扩容相关信息。
```go
// A header for a Go map.
type hmap struct {
    // 元素个数，调用 len(map) 时，直接返回此值
	count     int
	flags     uint8
	// B 是 buckets 数组的长度的对数，也就是说 buckets 数组的长度就是 2^B。buckets 里面存储了 key 和 value
	B         uint8
	// overflow 的 bucket 近似数
	noverflow uint16
	// 计算 key 的哈希的时候会传入哈希函数
	hash0     uint32
    // 指向 buckets 数组，大小为 2^B
    // 如果元素个数为0，就为 nil
	buckets    unsafe.Pointer
	// 等量扩容的时候，buckets 长度和 oldbuckets 相等
	// 双倍扩容的时候，buckets 长度会是 oldbuckets 的两倍
	oldbuckets unsafe.Pointer
	// 指示扩容进度，小于此地址的 buckets 迁移完成
	nevacuate  uintptr
	extra *mapextra // optional fields
}
```
- **bmap**：每个桶是 `bmap` 结构体，包含最多 8 个 `key-value` 对，如果超过这个数量，使用 `overflow` 字段指向下一个桶。
```go
type bmap struct {
    topbits  [8]uint8
    keys     [8]keytype
    values   [8]valuetype
    pad      uintptr
    overflow uintptr
}
```
- **扩容**：当桶满时，map 会进行扩容，新的桶数组会替换原有的桶数组。

- Go 的 `map` 使用链表来解决哈希冲突，每个桶存储若干 `key-value` 对，如果一个桶满了，会创建新的桶并通过 `overflow` 字段链接。
- `bmap` 存储桶内的实际数据。为了节省内存，Go 将 `key` 和 `value` **分别存储**，而不是交替存储，这样可以减少内存对齐时的填充字节。

##### **map 和 slice 的区别**：
- map 在传递到函数时，**传递的是指针**，因此对 map 的修改会影响原始数据。而 slice 是通过值传递的，修改传入的 slice 不会影响原始 slice。
- 一个是指针（`*hmap`），一个是结构体（`slice`）。Go 语言中的函数传参都是值传递，在函数内部，参数会被 copy 到本地。`*hmap`指针 copy 完之后，仍然指向同一个 map，因此函数内部对 map 的操作会影响实参。而 slice 被 copy 后，会成为一个新的 slice，对它进行的操作不会影响到实参。
##### **哈希函数**：
- Go 根据系统的 CPU 是否支持 AES 来选择哈希函数。如果支持 AES，使用 AES 哈希；否则，使用其他哈希算法。
##### **key 定位**：
- 对于 map 中的 `key`，哈希计算后得到一个 64 位的哈希值，通过该值来定位对应的桶。哈希值进一步决定桶内具体的位置。
**动态扩容**：
- `map` 的底层实现会动态调整哈希桶（buckets）数量。
- 无法直接获取或控制容量，但可以通过 `make(map[KeyType]ValueType, n)` 提示初始容量。
**触发扩容的条件**：
- 当元素数量增加到一定程度，触发扩容。
- 扩容会重新分配哈希桶并将现有键值对重新哈希（rehashing）。
**空间释放**
- `map` 删除键值对后，其占用的内存不会立即释放，但可以被垃圾回收。
- 如果需要完全释放 `map`，需要将其置为 `nil`

### 接口
鸭子类型，是动态编程语言的一种对象推断策略，它更关注对象能如何被使用，而不是对象的类型本身。Go 语言作为一门静态语言，通过接口的方式完美支持鸭子类型。静态语言在编译期间就能发现类型不匹配的错误，不像动态语言，必须要运行到那一行代码才会报错。
Go 采用了折中的做法：不要求类型显示地声明实现了某个接口，只要实现了相关的方法即可，编译器就能检测到。实际上是 Go 的编译器在其中作了隐匿的转换工作。
动态语言的特点：
> 变量绑定的类型是不确定的，在运行期间才能确定 函数和方法可以接收任何类型的参数，且调用时不检查参数类型 不需要实现接口

#### 方法
和函数的区别在于方法有一个接收者，给一个函数添加一个接收者，那么它就变成了方法。接收者可以是`值接收者`，也可以是`指针接收者`
不管方法的接收者是什么类型，该类型的值和指针都可以调用，不必严格符合接收者的类型。
实际上，当类型和方法的接收者类型不同时，其实是编译器在背后做了一些工作，用一个表格来呈现：

|         | 值接收者                                                     | 指针接收者                                                     |
| ------- | -------------------------------------------------------- | --------------------------------------------------------- |
| 值类型调用者  | 方法会使用调用者的一个副本，类似于“传值”                                    | 使用值的引用来调用方法，上例中，`qcrao.growUp()` 实际上是 `(&qcrao).growUp()` |
| 指针类型调用者 | 指针被解引用为值，上例中，`stefno.howOld()` 实际上是 `(*stefno).howOld()` | 实际上也是“传值”，方法里的操作会影响到调用者，类似于指针传参，拷贝了一份指针                   |
实现了**接收者是值类型的方法，相当于自动实现了接收者是指针类型的方法**；而实现了**接收者是指针类型的方法，不会自动生成对应接收者是值类型的方法**。
接收者是指针类型的方法，很可能在方法中会对接收者的属性进行更改操作，从而影响接收者；而对于接收者是值类型的方法，在方法中不会对接收者本身产生影响。
##### 何时使用
使用指针作为方法的接收者的理由：
- 方法能够修改接收者指向的值。
- 避免在每次调用方法时复制该值，在值的类型为大型结构体时，这样做会更加高效。
是使用值接收者还是指针接收者，不是由该方法是否修改了调用者（也就是接收者）来决定，而是应该基于该类型的`本质`。
- 当其成员都是由 Go 语言里内置的原始类型，如字符串，整型值等，那就定义值接收者类型的方法。像内置的引用类型，如 slice，map，interface，channel，这些类型比较特殊，声明他们的时候，实际上是创建了一个 `header`， 对于他们也是直接定义值接收者类型的方法。这样，调用函数时，是直接 copy 了这些类型的 `header`，而 `header` 本身就是为复制设计的。
-  如果类型具备非原始的本质，不能被安全地复制，这种类型总是应该被共享，那就定义指针接收者的方法。
### 并发相关
#### Go 的并发模型是什么？用 Goroutine 和 Channel 的优势是什么？
- Go 使用 **CSP 并发模型**（Communicating Sequential Processes），通过 Goroutines 和 Channels 实现。其核心思想是通过 **消息传递** 来进行并发编程，而不是通过共享内存进行数据交换。**注意与GMP调度模型区分**。
##### **CSP 的核心思想**
1. **进程是独立的：**  
    系统中的每个进程都是独立的，它们之间不共享内存，每个进程只能通过通信来协作。
2. **消息传递：**  
    不同的进程通过消息（Message）交换数据，而不是通过访问共享变量。
3. **同步通信：**  
    发送消息的进程和接收消息的进程需要在通信时刻进行同步（可以实现异步传递，但同步是基本模式）。
4. **组合复杂行为：**  
    通过对多个简单进程的组合，构建复杂的并发行为。
##### **优势**：
    - Goroutines 是go语言并发单元，是独立的轻量级线程，通过channel通信而非共享内存。
    - Channels 提供线程安全的通信方式，避免直接使用共享内存。
##### CSP在go中的关键特性
- 避免共享内存，通过通信传递数据
- 同步特性：无缓冲 Channel 的同步特性使得 Goroutine 之间能够进行精确的协调。
- **选择语句（`select`）：**  Go 提供了 `select` 语句来处理多个 Channel 的通信，从而支持非阻塞操作（default）或多路复用。time.After进行超时控制。
#### Go 运行时（Runtime）
##### 核心功能
- **Goroutine 调度**：实现了 **M:N 调度模型**，将 N 个 Goroutine 映射到 M 个内核线程上执行：
	- **G（Goroutine）：** 表示具体执行的任务。
	- **M（Machine）：** 操作系统线程，实际承担 Goroutine 的执行。
	- **P（Processor）：** 调度 Goroutine 的上下文，负责将 Goroutine 绑定到线程上。
- [[#GC垃圾回收]]（Garbage Collection，GC）
- 内存分配
- 并发原语支持
- 异常处理

**特点：**

- Goroutine 是用户态的轻量级线程，切换无需操作系统参与。
- 通过 `work stealing` 算法实现高效调度，动态分配 Goroutine 到线程。

##### goroutine队列相关
在 Go 的调度器（Go Scheduler）中，队列是管理和调度 **Goroutine** 的核心部分。Go 调度器使用多级队列来管理 Goroutine 的调度过程，以便高效地将 Goroutine 映射到操作系统线程（M）上执行。队列系统确保 Go 的并发调度在高并发和多核环境下能够高效运作。
以下是对 Go 调度器中 **队列** 相关机制的详细解析：
######  **1. Go 调度器中的主要队列**
Go 调度器使用了多个队列来管理 Goroutine 的调度，包括：
- **Goroutine 队列**：用于存储待执行的 Goroutine。
- **本地队列（Local Queue）**：每个 P（处理器）都有一个本地队列，存储分配给该 P 的 Goroutine。
- **全局队列（Global Queue）**：全局队列存储那些没有被本地队列处理的 Goroutine。
- **抢占队列（Preempted Queue）**：用于存储被抢占的 Goroutine，它们会被调度器重新调度执行。
###### **2. Goroutine 和队列的关系**
- **Goroutine** 是 Go 并发的基本单位，当我们通过 `go` 关键字启动一个 Goroutine 时，它会被加入到队列中等待调度。
- **调度器**（Scheduler）负责选择待执行的 Goroutine，将其分配给可用的 M 执行。
---
###### **3. 本地队列 (Local Queue)**
每个 **P**（Processor）都有一个本地队列，用来存储该 P 上待执行的 Goroutine。
**特点**：
- 本地队列是 Go 调度器的基本调度单元，每个 P 都会从本地队列中获取待执行的 Goroutine。
- 本地队列的目的是减少锁的竞争，提高 Goroutine 的调度效率。（因为访问全局队列需要加锁）
- 每个本地队列使用 **FIFO**（先入先出）顺序来存储 Goroutine。
**工作流程**：
1. 当一个 Goroutine 被创建时，它会被放入一个特定的 P 的本地队列。
2. 该 P 上的 M（操作系统线程）会从本地队列中取出 Goroutine 执行。
3. 如果本地队列为空，P 会去 **全局队列** 中窃取 Goroutine 来执行。
---
###### **4. 全局队列 (Global Queue)**
全局队列用于存储那些没有被分配到本地队列的 Goroutine。它是多个 P 共享的队列。
**特点**：
- 全局队列用于处理某些无法立即分配到本地队列的 Goroutine，通常用于协调各个 P 之间的调度。
- 每个 P 都可以从全局队列中窃取任务，避免某些 P 处于空闲状态。
**工作流程**：
1. 如果本地队列为空，P 会尝试从全局队列中 **窃取**（work stealing）任务。
2. 全局队列是线程安全的，多个 P 可以同时访问该队列。
---
###### **5. 抢占队列 (Preempted Queue)**
Go 调度器使用抢占机制来确保 **公平调度**，当一个 Goroutine 执行时间过长，或者被系统标记为需要抢占时，它会被移到抢占队列中。
**特点**：
- 被抢占的 Goroutine 会被放入抢占队列，这些 Goroutine 会稍后重新被调度执行。
- 抢占机制是 Go 调度器的一个重要特性，它保证了没有某个 Goroutine 会一直占用 CPU 时间，避免了长时间运行的 Goroutine 阻塞其他任务。
**工作流程**：
1. 当 Goroutine 执行超过一定时间（由调度器内的时间片控制）时，调度器会 **强制抢占**。
2. 被抢占的 Goroutine 会被移动到抢占队列中，等待下一次调度。
3. 调度器会重新将这些 Goroutine 分配给其他 M 执行。
---
###### **6. 调度流程中的队列操作**
Go 调度器会通过以下几步操作来决定哪个 Goroutine 应该执行：
1. **Goroutine 创建**：  
    当创建一个 Goroutine 时，它会被放入当前 P 的本地队列中。
2. **Goroutine 执行**：  
    如果本地队列中有任务，P 会从本地队列中取出 Goroutine 执行。如果本地队列为空，P 会尝试从全局队列中窃取任务。
3. **工作窃取**：  
    如果某个 P 没有任务执行，Go 调度器会通过工作窃取技术让它从其他 P 的本地队列中获取任务。
4. **抢占与迁移**：  
    当一个 Goroutine 执行时间过长时，调度器会根据抢占机制把它移动到抢占队列，等待其他 M 执行。
5. **Goroutine 完成**：  
    当 Goroutine 执行完成后，相关的内存和资源会被释放。
---
###### **7. 优化和调度策略**
- **工作窃取**：  
    工作窃取是 Go 调度器优化的核心之一，目的是减少 CPU 核心空闲的时间。当某个 P 空闲时，它会从其他 P 中窃取任务，这样可以平衡各个 CPU 核心的负载。
- **抢占式调度**：  
    Go 的调度器并非完全合作式的（即 Goroutine 自愿放弃 CPU），而是会采用抢占式调度。在每个 Goroutine 执行一定时间后，它可能会被调度器抢占并被移到抢占队列中，以保证系统不会被某个长期运行的 Goroutine 阻塞。
- **多级队列管理**：  
    Go 调度器管理多个队列（本地队列、全局队列和抢占队列），通过这些队列的多级管理，保证了 Goroutine 的调度既高效又公平。

Go 调度器的 **GMP**（**Goroutine**、**Machine** 和 **Processor**）是 Go 并发模型的核心部分，负责调度和管理 Goroutine 的执行。在 Go 运行时，**GMP** 模型通过 M:N 调度的方式来调度大量 Goroutine 在有限的操作系统线程上执行，确保高效的并发处理。
在这里，我将更加细致地讲解 **GMP** 模型，并区分每个部分的作用与关系。
###### **1. G (Goroutine)**

**Goroutine**（简称 **G**）是 Go 程序中的并发执行单位。它类似于操作系统中的线程，但比线程更轻量。

- **Goroutine 的创建**：  
    在 Go 中，通过 `go` 关键字来创建 Goroutine。当调用 `go f()` 时，`f()` 函数会作为一个 Goroutine 在后台并发执行。每个 Goroutine 都是由 Go 运行时调度器管理的，调度器负责决定何时、在哪个操作系统线程上执行该 Goroutine。
    
- **Goroutine 的栈**：  
    每个 Goroutine 都有独立的栈，通常初始栈大小为 **2KB**，这比操作系统线程的小得多。Go 会根据需要动态调整 Goroutine 的栈大小，这样能有效节省内存。
    
- **Goroutine 状态**：  
    Goroutine 会经历不同的状态，如 **等待**、**运行中**、**阻塞**（例如 I/O 阻塞）等，Go 调度器会根据这些状态来决定是否切换执行。
    
- **Goroutine 的管理**：  
    调度器负责将 Goroutine 放入队列（如本地队列、全局队列等），并选择适合的执行环境（P 和 M）来执行它。
---
###### **2. M (Machine)**

**M** 代表 **Machine**，即操作系统线程。Go 调度器使用多个 M 来执行 Goroutine，每个 M 都对应一个操作系统线程。M 的数量通常是系统核心数（CPU 核心数）的倍数，但不会无限制增长。

- **M 的功能**：
    
    - M 是操作系统线程，执行具体的工作。每个 M 会从与之关联的 P 上的队列中获取 Goroutine 来执行。
    - 每个 M 都是与操作系统线程直接挂钩的，因此它会被操作系统调度到不同的 CPU 核心上。
- **M 的调度**：
    
    - 每个 M 与 P 绑定，它从 P 的本地队列中获取 Goroutine 执行。当 P 的队列为空时，M 会去从其他 P 的队列窃取任务（工作窃取，Work Stealing）。
    - M 的数量是由 Go 调度器管理的，通常会根据系统的 CPU 核心数来调整。Go 运行时会根据当前并发需求动态调整 M 的数量。
- **M 的生命周期**：
    
    - M 会在执行完成一个 Goroutine 后，查看本地队列是否还有任务。如果本地队列为空，M 会尝试去其他 P 处偷任务，或者进入空闲状态，等待新的任务。
    - 如果没有任务可执行，M 会被回收，直到需要更多 M 来执行新的 Goroutine。

---

###### **3. P (Processor)**

**P** 代表 **Processor**，即 Go 调度器中的一个抽象单位，用来调度 Goroutine 执行。每个 P 上都有一个 **本地队列**，它存储待执行的 Goroutine。P 实际上是 Go 调度器的执行上下文，它决定哪些 Goroutine 会在 M 上运行。

- **P 的功能**：
    
    - P 是执行 Goroutine 的上下文环境，每个 P 都会维护一个本地队列，存储该 P 上待执行的 Goroutine。
    - Go 调度器会把 M 绑定到 P 上，每个 P 最多只能执行一个 M。
- **P 与 M 的关系**：
    
    - 每个 P 上会绑定一个 M 来执行该 P 上的 Goroutine。一个 M 可以在多个 P 上执行，但每个 P 只能绑定一个 M。
    - P 控制 M 的调度，决定将哪个 M 与哪个 Goroutine 绑定执行。M 会在 P 上取任务并执行。
- **P 的数量**：
    
    - Go 调度器会根据可用 CPU 核心数来决定 P 的数量。每个 P 对应一个 CPU 核心。
    - P 的数量决定了并发的粒度，如果 P 的数量少于可用 CPU 核心数，Go 调度器会通过工作窃取机制来平衡负载。

---

###### **4. GMP 之间的关系与调度流程**

#### **1. Goroutine 的调度**

- 当你通过 `go` 启动一个新的 Goroutine 时，该 Goroutine 会被加入到某个 P 的 **本地队列**中。
- 每个 P 上的 **M** 负责从本地队列中获取 Goroutine 执行。
- 如果 P 上没有 Goroutine，M 会尝试从其他 P 上的队列中 **窃取任务**（Work Stealing），确保所有的 Goroutine 都能被执行。

#### **2. M、P、G 之间的工作流程**

1. **创建 Goroutine**：
    - 创建 Goroutine 时，它会被加入到一个特定的 P 上的队列中。
2. **调度 Goroutine**：
    - M 从 P 的本地队列中获取一个 Goroutine 并执行。
    - 如果本地队列为空，P 会去全局队列或其他 P 的队列窃取任务。
3. **执行 Goroutine**：
    - M 执行 Goroutine 后，检查是否有新的任务。如果没有，它会与 P 一起处于空闲状态，直到更多任务到达。
4. **抢占和调度**：
    - Go 调度器使用抢占式调度，如果一个 Goroutine 执行时间过长，调度器会强制抢占它，移到抢占队列，等待其他 M 来执行。

#### **3. 工作窃取（Work Stealing）**

- 当某个 P 的本地队列为空时，P 会从其他 P 的队列中窃取任务。这种机制确保了多核 CPU 系统下的负载平衡，避免某些核心一直空闲。

#### **4. P 的空闲管理**

- 每个 P 会管理一个本地队列，当队列为空时，P 会通过工作窃取机制与其他 P 协作获取任务。P 的数量通常和系统的 CPU 核心数相同，以保证多核高效利用。

---

###### **5. 总结：GMP 的区别与配合**

- **G (Goroutine)**：Go 程序中的基本并发单元，轻量级的线程，负责执行实际的任务。
- **M (Machine)**：操作系统线程，负责在操作系统层面执行 Goroutine，M 可以在不同的 P 上执行 Goroutine。
- **P (Processor)**：调度上下文，负责调度 Goroutine 执行。每个 P 对应一个 CPU 核心，并维护一个本地队列。

通过这种 **M:N** 调度模型，Go 实现了轻量级的并发，能够在多个 CPU 核心上高效地调度和执行大量 Goroutine，同时保证调度的高效性和公平性。

###### 1. **M（Machine）和 P（Processor）的定义**

- **M** 代表操作系统线程，它与实际的操作系统线程挂钩，是执行任务的基本单位。
- **P** 代表调度器的“处理器”，它是一个虚拟的概念，用来调度和执行 Goroutine。每个 **P** 都有一个本地队列，用于存储待执行的 Goroutine。

###### 2. **为什么一个 M 可以在多个 P 上执行？**

在 Go 中，一个 **M** 可以在多个 **P** 上执行是因为：

- **M 是操作系统线程**，它并不是 Go 调度器中的一个固定角色。每个 **M** 是与操作系统线程绑定的，操作系统线程（或 M）是能够在不同的 CPU 核心上执行任务的。操作系统会将线程调度到不同的 CPU 核心上，而不一定固定在一个 CPU 核心上。
- **Go 的调度器是基于 M:N 模型**，意味着多个 Goroutine 会被调度到较少数量的操作系统线程上运行。为了充分利用多核处理器，M 可以从不同的 **P** 获取任务并执行。
- 如果一个 **M** 在某个 **P** 上执行完任务后，**P** 的队列为空，**M** 会被调度到另一个 **P** 上，去获取新的任务。这使得 **M** 可以动态地在多个 **P** 上执行任务，从而提高 CPU 的利用率。

#### 举个例子：

- 假设系统有 4 个 **P**（每个 **P** 对应一个 CPU 核心），并且有 10 个 **M**。每个 **M** 可以在不同的 **P** 上执行，假如一个 **M** 在 **P1** 上执行完任务，**P1** 的队列为空时，**M** 可以去 **P2** 或 **P3** 的队列里窃取任务，继续执行。这意味着一个 **M** 可以跨多个 **P** 上执行任务，从而保证并发任务的平衡。

###### 3. **为什么每个 P 只能绑定一个 M？**

每个 **P** 只能绑定一个 **M** 的原因与 Go 调度器如何管理资源和确保高效执行有关：

- **P 是调度上下文**，它负责管理任务的调度，每个 **P** 维护着一个本地队列，存储待执行的 Goroutine。为了避免并发调度中的复杂性，**每个 P 只能绑定一个 M**，这样可以确保调度过程简单且高效。如果每个 **P** 同时绑定多个 **M**，就会导致调度过程中的同步问题，增加调度器的开销。
- **避免多 M 同时操作同一队列**：每个 **P** 的本地队列由绑定的 **M** 来执行。如果一个 **P** 同时绑定多个 **M**，会导致多个线程（操作系统线程）同时访问同一队列，这就需要复杂的锁和同步机制来避免竞态条件（race condition），从而降低了性能。
- **每个 P 有一个独立的调度上下文**，且每个 **M** 被调度到 **P** 上时，M 只负责从 P 的队列中获取并执行 Goroutine。这使得每个 **P** 的调度过程更加清晰、独立，有助于减少竞争。

###### 4. **总结**

- **一个 M 可以在多个 P 上执行**，因为 M 是操作系统线程，它不依赖于特定的 P。M 会在系统中动态地移动，调度任务来执行，并且可以跨多个 P 执行任务以确保 CPU 的高效利用。
- **每个 P 只能绑定一个 M**，是为了避免调度过程中的复杂性和竞争，确保每个 P 的调度上下文简单且高效，避免多个线程同时访问同一个队列，从而减少同步开销。

这种设计确保了 Go 在处理大量 Goroutine 时的调度效率，并能够充分利用多核 CPU，提高并发性能。

### 反射
reflect实现了运行时的反射能力，能够让程序操作不同类型的对象。两对重要的函数和类型：
- `reflect.TypeOf`能获取类型信息；`reflect.Type`
- `reflect.ValueOf` 能获取数据的运行时表示；`reflect.Value`
类型 `reflect.Type` 是反射包定义的一个接口，是TypeOf返回的任意变量类型；接口中定义了一些有趣的方法：`MethodByName` 可以获取当前类型对应方法的引用、`Implements` 可以判断当前类型是否实现了某个接口
```go
type Type interface {
        Align() int
        FieldAlign() int
        Method(int) Method
        MethodByName(string) (Method, bool)
        NumMethod() int
        ...
        Implements(u Type) bool
        ...
}
```
反射包中 `reflect.Value` 的类型与 `reflect.Type` 不同，它被声明成了结构体。这个结构体没有对外暴露的字段，但是提供了获取或者写入数据的方法：
```go
type Value struct {
        // 包含过滤的或者未导出的字段
}

func (v Value) Addr() Value
func (v Value) Bool() bool
func (v Value) Bytes() []byte
```
反射包中的所有方法基本都是围绕着 `reflect.Type` 和 `reflect.Value` 两个类型设计的。我们通过 `reflect.TypeOf`、`reflect.ValueOf` 可以将一个普通的变量转换成反射包中提供的 `reflect.Type` 和 `reflect.Value`，随后就可以使用反射包中的方法对它们进行复杂的操作。

过量的使用反射会使我们的程序逻辑变得难以理解并且运行缓慢

	1. 三次握手：建立可靠全双工连接
		1. **SYN**：**客户端**发送一个 **SYN**（同步）包给服务器，表示客户端希望建立连接，并告知**客户端的初始序列号 `seq`**。此时客户端进入 `SYN_SENT` 状态。
		2. **SYN-ACK**：服务器收到客户端的 **SYN** 包后，回复一个 **SYN-ACK** 包，表示服务器已准备好与客户端建立连接，确认客户端的序列号，同时发送服务器的初始序列号 `ack`。此时，服务器进入 `SYN_RECEIVED` 状态
		3. **ACK**：客户端收到服务器的 **SYN-ACK** 包后，发送一个 **ACK**（确认）包给服务器，表示客户端已经接收到服务器的确认，同时确认服务器的序列号。此时，客户端进入 `ESTABLISHED` 状态，服务器也进入 `ESTABLISHED` 状态。
	2. 四次挥手：确保连接能够安全关闭，防止数据丢失。
		1. **FIN**：客户端发送一个 **FIN**（结束）包给服务器，表示客户端没有数据要发送了，并希望关闭连接。客户端进入 `FIN_WAIT_1` 状态。
		2. **ACK**：服务器收到客户端的 **FIN** 包后，回复一个 **ACK** 包，表示确认收到客户端关闭连接的请求，服务器进入 `CLOSE_WAIT` 状态。
		3. **FIN**：服务器在确认完客户端的 **FIN** 后，准备关闭连接，向客户端发送 **FIN** 包，表示服务器没有数据要发送，并希望关闭连接。服务器进入 `LAST_ACK` 状态。
		4. **ACK**：客户端收到服务器的 **FIN** 包后，发送一个 **ACK** 包给服务器，确认收到服务器的关闭连接请求。此时，客户端进入 `TIME_WAIT` 状态，并等待 2MSL（最大报文生存时间，通常是 2 倍的最大报文寿命）后彻底关闭连接。服务器收到客户端的 **ACK** 包后，进入 `CLOSED` 状态。
