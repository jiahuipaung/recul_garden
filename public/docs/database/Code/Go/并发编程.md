# 线程与进程

并发不是并行。只有说将go runtime配置为多个逻辑处理器时，逻辑处理器可能可以绑定在一个线程上，线程在**不同物理处理器**上运行，这样才能实现真正的并行。否则goroutine在同一物理处理器上运行，即使设置了多个逻辑处理器


线程的创建与销毁，成本与门槛都很高，上下文切换，唤醒与睡眠的代价都很高

空select语句永远阻塞

- 如果当前goroutine需要等其他goroutine完成才能继续进行，那么该任务最好你自己完成：keep yourself busy or do the work yourself
- 让调用者决定是否要并发
	- 例如listDirectory的slice和channel两种实现，channel进行了并发但是调用者对此感受不直接，且对错误/结束的感知具有二义性
	- 因此需要将后台函数是否继续执行的决定权赋给调用方

使用goroutine时思考两个问题：
1. 这个goroutine什么时候会结束
	1. 可以使用log.Fatal使goroutine在异常时退出，但是由于调用了os.Exit来不急执行defer。（因此Fatal只适合在init函数中和main中调用）
2. 调用者是否有手段可以结束该goroutine
3. 也就是说调用者需要能够管理goroutine的生命周期

# Memory Model
跨goroutine读取其他goroutine中的数据，如果修改数据时有其他语句同步在读取，则必须串行化，使用channel或其他同步原语
Go 内存模型指定了这样一个条件，在这个条件下，可以保证对一个协程中变量的读取可以观察到对不同协程中同一变量的写入所产生的值。

CPU优化
- 多核心的情况下，无法预测两断程序是否是等价的；
- store buffer机制，也会导致多核心时结果不可预测
由于这种情况存在，才要求使用CPU提供的锁机制，要求所有对内存的操作都必须扩散到Memory之后才能继续进行对其他memory的操作
