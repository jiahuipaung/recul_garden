# 了解标准库errors包
errors.New()返回内部errorstring对象的指针。这种方法当返回两个相同错误时不会进行字符串匹配的判断，避免一些意外情况
errors.New("包名：错误内容")

多返回值返回了error时，必须要先处理error，唯一可以忽略error的是如果你连value也不关心

go中的panic与其他语言中的exception不同，exception抛出会给调用者来处理，但是panic并不会有人来处理，完全game over。因此你知道什么时候有良性错误可以解决，什么时候是异常。recover只能作为兜底，不能作为try-catch模式。

因此对于真正的意外，不可恢复的程序错误：索引越界，不可恢复的环境问题，栈溢出才使用panic
对于其他错误情况用error判定和处理，

## Error types
- **实现了error接口**的自定义类型
- 再对err进行类型断言，是否是自己实现的MyError。相比Sentinel error能通过封装底层error提供更多上下文信息
- 由于需要进行类型断言和switch，因此也是public，调用者会与其产生强耦合
## Opaque errors
即判断返回的err != nil。只返回错误而不假设其内容
而为了应对相应错误，我们其实不需要错误的类型，而是需要错误的行为。因此断言错误实现了特定的行为。而不是断言特定的类型和值

## wrap errors
直接层层上抛error时没有报错信息，需要分析到底是哪里报错的，没有行号和上下文信息
最佳实践：一个error只处理一次，要么上抛，要么打日志，层层上抛的过程就不应该一直打日志
- 日志记录与错误无关，且对调试没有任何作用，就是噪音
- 日志记录的原因是某些东西失败了，而日志包含答案。后续不应该再报告当前错误
- 处理错误必须保证100%的完整性


# errors使用技巧
- errors目前 可以通过wrap包装上下文，再通过Cause获取错误根因，人和机器都可以使用
- 在业务代码中自身原因出现错误，使用errors.New和errors.Errorf返回错误，可以携带原始的堆栈信息
- 如果调用其他包内的函数就直接返回往上抛
- 如果和其他库（第三方库）进行协作，考虑使用wrap或wrapf报错堆栈信息，包括标准库。库报错时。但是第三方库自身不应该已经使用wrap，否则会有两坨冗余堆栈信息。只应该在application中使用
- 能处理就处理，处理不了就往上抛。带上有用的上下文
- 直接返回错误，而不是到处打日志
- 如果一个错误已经被处理过，就不应该再往上抛，需要返回的话就只能返回nil
- 在程序顶部或者工作的goroutine顶部，请求入口，使用%+v来记录堆栈
- 使用errors.Cause获取root error，在和sentinel error判定

## error inspecting
- 自定义的err，如果实现了Unwrap方法，就可以返回底层的error。可以不断调用unwrap来寻找根因进行匹配
- fmt.Errorf %w这种方法返回的wrapError就实现的unwrap，因此可以使用Is和As进行判定。但是没有包含堆栈信息
- pkg/errors的wrap包含了堆栈信息，并且兼容了go1.13的errors
